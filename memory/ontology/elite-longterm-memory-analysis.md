## 二、6 层记忆架构深度分析

### Layer 1: Hot RAM (SESSION-STATE.md)
- **核心能力**: 会话级热记忆，抗压缩、抗重启
- **关键协议**: WAL (Write-Ahead Log) — 先写，再回复
- **触发场景**: 用户表达偏好、做出决策、给出截止日期、纠正 Agent
- **边界**: 仅限当前会话范围，跨会话需要显式升级到上层

### Layer 2: Warm Store (LanceDB 向量)
- **核心能力**: 语义搜索、自动召回
- **触发场景**: 会话开始时自动检索相关历史
- **边界**: 依赖 OpenAI Embedding API，有调用成本
- **改进建议**: 考虑使用本地 Embedding 模型降低成本

### Layer 3: Cold Store (Git-Notes 知识图谱)
- **核心能力**: 永久结构化存储、分支感知
- **触发场景**: 重要技术决策、架构选择
- **边界**: 需要工作区是 Git 仓库，Git 命令可用

### Layer 4: Curated Archive (MEMORY.md + daily)
- **核心能力**: 人可读长期记忆、人工审核层
- **触发场景**: 每周回顾、月度总结
- **边界**: 需要人工审核保证质量，自动化程度较低

### Layer 5: Cloud Backup (SuperMemory) — 可选
- **核心能力**: 跨设备同步、知识库聊天
- **边界**: 需要第三方 API，数据离开本地

### Layer 6: Auto-Extraction (Mem0) — 推荐
- **核心能力**: 自动事实提取、80% Token 节省
- **边界**: 需要额外 NPM 包和 API Key

### 详细联动模式

| # | 联动组合 | 模式 | 说明 |
|---|----------|------|------|
| 1 | Elite Memory + Ontology | 分层架构 | Elite Memory 负责原始记录（文件存储），Ontology 负责结构化索引（实体/关系查询）。两者互补：Elite 保证写入速度和可读性，Ontology 提供高级查询能力。 |

## 四、最佳使用实践 (6 条)

| # | 实践 | 分类 | 说明 |
|---|------|------|------|
| 1 | **严格遵循 WAL 协议** | 可靠性 | Write FIRST, respond LATER. 任何状态变更先写入 SESSION-STATE，再回复用户。这是防止记忆丢失的最关键保障。 |
| 2 | **分层存储策略** | 性能 | Hot → Warm → Cold → Curated 四层层级，平衡速度、成本、质量。不是所有信息都需要永久存储，不是所有存储都需要可搜索。 |
| 3 | **定期记忆卫生** | 维护 | 每周执行记忆清理：删除过时向量、合并重复记录、精选到 MEMORY.md。保持记忆系统的信噪比。 |
| 4 | **最小写入原则** | 质量 | 只记录重要信息（决策、偏好、阻塞点），避免噪音淹没信号。人类记忆力有限是有道理的。 |
| 5 | **人工审核归档层** | 质量 | Curated Layer (MEMORY.md) 必须经过人工审核。自动化提取可能出错，人类的判断力是最后一道质量防线。 |
| 6 | **Antifragile 记忆** | 改进 | 每次错误都记录到 lessons.md。不要浪费任何一次失败，让系统从错误中学习变强。 |

## 五、已知边界与限制 (8 项)

1. **LanceDB 插件依赖** — 向量搜索需要额外安装插件
2. **OpenAI API Key 依赖** — Embedding 和自动提取需要有效 Key
3. **Windows 路径兼容问题** — CLI 使用 `process.env.HOME` 在 Windows 上可能 undefined
4. **无并发写入控制** — 多 Agent 同时写相同记忆文件可能冲突
5. **无内置版本控制** — 记忆文件无内置回滚机制（Git 仓库可间接解决）
6. **记忆维护需手动执行** — Weekly hygiene 需要手动触发，无自动调度
7. **Git 仓库依赖** — Git-Notes 功能需要工作区是 Git 仓库
8. **子代理记忆隔离** — Sub-agent 默认不继承父代理上下文，需要手动传递
